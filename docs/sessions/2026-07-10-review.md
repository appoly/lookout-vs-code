# Full-codebase review — 2026-07-10

> Historical review. Several findings—default labels, guided executable checks, guarded restart, provider enablement, unread badge/status, and updated review-artifact behavior—have since been implemented. Current status lives in [ROADMAP](../ROADMAP.md), and manual acceptance criteria live in [TESTPLAN](../TESTPLAN.txt).

A complete review of the extension (all source, tests, docs, and packaging) against three questions: what is strong, what falls short, and what is required for a Marketplace release. Findings were verified against `cf4bc9b Refine agent review and usage UX`; `npm run check` and `npm run vsix` were green throughout.

Two review findings were implemented the same day, before this record was written: session-only Codex `notify` turn-complete integration (now D8) and the neutral `active` process state distinct from a working turn (now part of D3).

## What is working well

- **Differentiated thesis.** "Orchestration layer over VS Code, not a terminal emulator in a webview" (D1) is the opposite of most competitors and compounds as VS Code improves: native TTY fidelity, `vscode.diff`, remoting, and themes come for free.
- **Honesty enforced in code, not just docs.** Six distinct usage statuses; restored sessions visibly report a dead bridge instead of pretending; lifecycle claims are limited to the exact execution Parful launched; shared-worktree changes are never attributed to one agent.
- **Security posture.** Loopback-only bridge with random bearer token, body limits, and timeouts; workspace-trust gating with `restrictedConfigurations`; custom commands never persisted; no credential files read; no terminal scraping.
- **Testable architecture.** Pure domain modules (`sessionModel`, `agentCommand`, `claudeUsage`, `gitReview` parsers, `usageFormatting`) are separated from VS Code glue, which is why real unit tests — including a live bridge test and a temp-git-repo test — run without an extension host.
- **Docs discipline.** DECISIONS/RESEARCH/ROADMAP/session checkpoints let a cold reader resume safely.

## Defects and gaps found

1. **Restart can type into a live agent.** `SessionManager.restart` sends the launch command into the existing terminal without checking `agentExecutions` for a still-running execution. Restarting a live interactive session delivers the command text to the agent's stdin as a prompt. It should refuse or confirm (and offer close-and-relaunch) while the tracked execution is running. *Confirmed present at `cf4bc9b`.*
2. **First-run experience assumes the author's context.** The label prompt is mandatory on every launch (should default and be skippable, with rename afterwards); a missing `codex`/`claude` executable surfaces as a shell error in a dead terminal instead of a guided message; hooks require `node` on the terminal's PATH with no check.
3. **Usage surfaces assume both providers.** The status bar permanently shows `Codex —` for Claude-only users, and `codex app-server` is spawned regardless. Per-provider enablement settings would quiet both.
4. **Attention is not glanceable when the sidebar is closed.** Attention lives in tree rows and transient toasts. An unread-count badge on the activity-bar icon (requires `createTreeView` instead of `registerTreeDataProvider`) is cheap and matches the cockpit promise.
5. **Namespace split was about to fossilize.** At review time, internal IDs were `multiTerm.*` while every visible string said Paraterm. This was resolved before publication by D10: the product and namespace are now Parful / `parful.*`.
6. **Minor, tracked, not blocking:** the shared generated Claude settings file is rewritten on every launch while older sessions still reference it (benign while content is deterministic; a trap for future edits); close has no safe-close flow yet (already on the roadmap); the md/mdx watcher triggers a full artifact refresh on every workspace Markdown save, which large workspaces may feel.

## Marketplace readiness

Ordered checklist:

1. **Prove it interactively.** Finish the remaining smoke items (provider live lifecycle, quota reads, browser, reload restoration, installed-VSIX path). The release gates already require this.
2. **Identity decisions (one-way doors).** Marketplace publisher account and `parful` name availability; make the repository public at the manifest URL; decide whether to rename the external GitHub repository; remove stale prototype VSIX files.
3. **Listing assets.** `icon` (128×128+ PNG — the SVG only covers the activity bar), `galleryBanner`, screenshots and a short GIF of the launch → attention → review loop, a user-first README rewrite (quickstart at top, development status and architecture notes demoted), the `AI` category, and agent-name keywords.
4. **Quality gates.** Extension-host tests via `@vscode/test-cli` (activation, commands, terminal creation, virtual baseline provider) on both VS Code 1.96 and stable; Windows and macOS CI legs (`shellQuote` and path logic have platform branches with no coverage there); an esbuild bundle step.
5. **Release automation.** Tag-driven `vsce publish` plus Open VSX publishing — much of the target audience is on Cursor/VSCodium/Windsurf, which do not use the Microsoft Marketplace.
6. **Ship 0.1 with `"preview": true`.**

## Innovation priorities

Ranked by differentiation on top of what exists:

1. **Worktree-per-agent launch.** Review grouping and honest attribution already exist; creating the isolation ("New Agent → in a fresh worktree": branch prompt, `git worktree add`, launch there) turns Parful into a genuine parallel-agent workflow and makes per-agent diffs true. Pair with the safe-close flow (running process → dirty worktree → keep/review/remove).
2. **Session templates.** Agent + flags + worktree policy + task + browser URL as a one-keystroke mission from the `+` menu.
3. **Attention inbox.** A chronological hook-event feed with next/previous triage on top of the existing `ctrl+alt+n`, plus the activity-bar badge.
4. **Broader agent roster.** Gemini CLI, opencode, Aider, Amp are each "a command plus maybe a hook adapter"; first-class entries multiply search surface and audience.
5. **Session resume** (`codex resume`, `claude --resume`) on restore — open decision 2.
6. **Diff stats in the review tree** (`git diff --numstat`: `+120 −14 across 6 files` per worktree).

## Recommended sequence

- **Phase 0 — prove it (blocking):** remaining smoke matrix in a desktop VS Code; fix the restart guard (finding 1).
- **Phase 1 — credible for strangers:** activity-bar badge, per-provider usage enablement, first-run polish (skippable label, executable detection), decide the namespace.
- **Phase 2 — Marketplace launch:** extension-host tests + CI matrix + bundling, listing assets and user-first README, publisher + tag-driven publish + Open VSX, ship 0.1 as preview.
- **Phase 3 — differentiator:** worktree-per-agent + safe close, then templates, resume, inbox, roster — sequenced by early-user demand.

Do not start Phase 3 before Phase 0 passes: the project's risk is unverified runtime behavior, not code quality.
